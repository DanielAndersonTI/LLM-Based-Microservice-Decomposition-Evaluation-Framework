# LLM-Based Microservice Decomposition Evaluation Framework

[![License](https://img.shields.io/badge/license-MIT-blue.svg)](LICENSE)
[![Python](https://img.shields.io/badge/python-3.8%2B-blue)](https://www.python.org/)
[![Research](https://img.shields.io/badge/research-UFCG-green)](https://www.ufcg.edu.br/)

An automated framework for evaluating the **structural adherence of microservice architectures generated by Large Language Models (LLMs)** against the actual source-code structure of monolithic systems.

This project was developed as part of a master's research project at the **Federal University of Campina Grande (UFCG)**. The research investigates how accurately LLM-generated microservice decompositions reflect the structural organization of real software systems.

# Research Questions

This research is guided by the following questions:

- **RQ1:** What is the degree of structural adherence between architectures generated by LLMs and the actual source code of a monolithic system?

- **RQ2:** Does the LLM model used (e.g., OpenAI o3, Claude 3.5 Sonnet, or future models) influence the structural adherence of the generated decomposition?

- **RQ3:** Do prompting strategies (Zero-Shot and Few-Shot) impact the quality of the resulting microservice architectures?

# Framework Overview

The framework consists of four sequential modules:

```text
┌─────────────────────┐    ┌─────────────────────┐    ┌─────────────────────┐    ┌─────────────────────┐
│ Dependency          │    │ Architecture        │    │ Mapping and         │    │ Metrics and         │
│ Extraction          │ →  │ Interpretation      │ →  │ Violation Detection │ →  │ Summarization       │
│ (Real Graph)        │    │ (LLM Outputs)       │    │                     │    │                     │
└─────────────────────┘    └─────────────────────┘    └─────────────────────┘    └─────────────────────┘
         │                         │                           │                           │
         ▼                         ▼                           ▼                           ▼
   Source Code              CSV Decompositions         Real Graph ×              TVD, SPR and
   → GraphML Graph          Generated by LLMs          Proposed Architecture     Granularity
```

## Module 1 — Dependency Extraction

Parses the source code of a monolithic application using Tree-sitter and extracts structural dependencies.

Extracted relationships include:

- Inheritance
- Interface implementation
- JPA relationships
- Composition
- Dependency injection
- Repository usage

Output:

- GraphML dependency graph

---

## Module 2 — Architecture Interpretation

Loads CSV files generated by LLMs and maps classes to proposed microservices.

Example:

```csv
class,service
Owner,Client Service
OwnerController,Client Service
Pet,Pet Service
```

---

## Module 3 — Mapping and Violation Detection

Cross-references the dependency graph with the proposed architecture.

For each dependency:

```text
A → B
```

the framework verifies:

- Same service → consistent dependency
- Different services → structural violation

---

## Module 4 — Metrics and Summarization

Computes the final evaluation metrics.

| Metric | Description | Formula |
|----------|-------------|----------|
| TVD (Dependency Violation Rate) | Proportion of dependencies crossing service boundaries | violations / total dependencies |
| SPR (Service Purity Rate) | Proportion of dependencies internal to services | 1 − TVD |
| Granularity | Number of proposed services | service count |

---

# Metric Naming Note

During the Spring PetClinic pilot study, the metric currently called **SPR (Service Purity Rate)** was originally referred to as:

```text
CB (Basic Cohesion)
```

The metric definition remains exactly the same.

| Historical Name | Current Name |
|----------------|-------------|
| CB (Basic Cohesion) | SPR (Service Purity Rate) |

Formula:

```text
CB = SPR = 1 − TVD
```

The terminology was updated to improve clarity and avoid confusion with traditional software cohesion metrics.

# How to Run

## 1. Dependency Extraction

```bash
cd Scripts
python petclinic_static_analysis.py
```

This step:

1. Parses all Java source files
2. Extracts dependencies
3. Builds the dependency graph
4. Exports GraphML

---

## 2. Graph Audit

```bash
python graph_audit.py
```

This step:

1. Creates timestamped backups
2. Exports graph edges
3. Computes node degrees
4. Detects isolated nodes
5. Generates an audit report

---

## 3. Structural Violation Detection

```bash
python violation_detection.py
```

This step:

1. Loads the dependency graph
2. Loads LLM decompositions
3. Detects structural violations
4. Computes TVD, SPR, and Granularity
5. Generates Markdown and JSON reports

# Pilot Study: Spring PetClinic

The Spring PetClinic system was used as a pilot study to validate and refine the framework.

## Pilot Metrics

| Metric | Few-Shot | Zero-Shot |
|----------|----------|----------|
| TVD | 20.00% | 22.50% |
| SPR | 80.00% | 77.50% |
| Granularity | 8 Services | 8 Services |

## Main Findings

- Few-Shot prompting achieved better structural adherence.
- Automatic dependency extraction was validated.
- Violation detection successfully identified cross-service dependencies.
- The framework proved capable of evaluating architecture proposals generated by LLMs.

# Adapting the Framework

## Java Projects

Modify:

```python
PETCLINIC_ROOT = Path("path/to/src/main/java")
```

Then execute:

```bash
python petclinic_static_analysis.py
python graph_audit.py
python violation_detection.py
```

---

## Other Languages

Install the appropriate Tree-sitter grammar.

Examples:

```bash
pip install tree-sitter-python
pip install tree-sitter-javascript
pip install tree-sitter-c-sharp
pip install tree-sitter-go
```

Update parser initialization accordingly.

---

## Custom LLM Outputs

Provide CSV files with the format:

```csv
class,service
ClassA,ServiceA
ClassB,ServiceB
```

Then update the input path in `violation_detection.py`.

# Limitations

- Static analysis only.
- Runtime dependencies are not captured.
- Reflection and dynamic bean creation are not modeled.
- Test code is excluded by default.
- The graph represents structural dependencies, not execution flow.
- Other programming languages require AST extraction adaptations.
- Class-name matching may occasionally generate false positives.
- The number of evaluated systems is still limited.

# Documentation Consistency

This repository contains historical artifacts generated during the pilot phase.

Some documents may still refer to:

```text
CB (Basic Cohesion)
```

while newer versions use:

```text
SPR (Service Purity Rate)
```

These terms refer to the exact same metric:

```text
CB = SPR = 1 − TVD
```

No recalculation of results is necessary.
