# Experimental Evaluation of the Divergence Between LLM-Generated Monolith Decompositions and the Actual Source Code Structure – A Pilot Study Using Spring PetClinic

## Description

This repository contains all artifacts required to reproduce a pilot experiment evaluating the structural adherence between microservice decompositions generated by Large Language Models (LLMs) and the actual source code structure of a monolithic system. The study uses **Spring PetClinic**, a widely adopted academic and industrial benchmark for software architecture research, as its experimental subject.

The evaluated decompositions were generated by **OpenAI o3 (GPT-4 o3)** using two prompting strategies: **zero-shot** and **few-shot**. The analysis pipeline automatically extracts structural dependencies from the source code, builds a dependency graph, and measures the alignment between the service boundaries proposed by the LLM and the dependencies actually present in the system. The evaluation is based on the metrics **Dependency Violation Rate (DVR)**, **Dependency Preservation Rate (DPR)**, and **Granularity**.

## Reproducing the Experiment

### 1. Obtain the Spring PetClinic Source Code

Clone the original Spring PetClinic repository into the `input/monolith/` directory:

```bash
cd input/monolith

git clone https://github.com/spring-projects/spring-petclinic.git spring-petclinic-main
```

The expected structure is:

```text
input/
└── monolith/
    └── spring-petclinic-main/
```

---

### 2. Run the Static Analysis

Execute:

```bash
python pipeline/petclinic_static_analysis.py
```

This script:

* Traverses the Spring PetClinic Java source code;
* Extracts classes, packages, and structural relationships;
* Identifies dependencies between components;
* Builds the system dependency graph;
* Exports the intermediate artifacts required by subsequent stages.

Generated files:

```text
intermediate/
├── petclinic_dependency_graph.graphml
└── petclinic_dependency_graph_edges.csv
```

---

### 3. Run the Graph Audit

Execute:

```bash
python pipeline/graph_audit.py
```

This script:

* Validates the integrity of the generated graph;
* Checks for structural inconsistencies;
* Produces descriptive graph statistics;
* Generates an audit report.

Generated file:

```text
intermediate/
└── petclinic_dependency_graph_audit.txt
```

---

### 4. Run Violation Detection

Execute:

```bash
python pipeline/violation_detection.py
```

This script:

* Loads the microservice groupings generated by the LLM;
* Compares the proposed service boundaries against the actual dependencies extracted from the source code;
* Detects dependencies crossing service boundaries;
* Computes the DVR, DPR, and granularity metrics;
* Generates the final analysis reports.

Generated files:

```text
results/
├── petclinic_violations_report.md
└── petclinic_violations_analysis.json
```

## Evaluated Metrics

### DVR — Dependency Violation Rate

Measures the proportion of dependencies that cross service boundaries.

Definition:

```text
DVR = Dependencies Crossing Service Boundaries
      -----------------------------------------
              Total Dependencies
```

Interpretation:

* Lower values are better.
* Low values indicate greater structural adherence to the system's actual architecture.

---

### DPR — Dependency Preservation Rate

Represents the proportion of dependencies preserved within the proposed service boundaries.

Definition:

```text
DPR = 1 − DVR
```

Interpretation:

* Higher values are better.
* High values indicate stronger alignment between the proposed decomposition and the structure observed in the source code.

---

### Granularity

Corresponds to the total number of services proposed by the decomposition.

Interpretation:

* Higher values indicate more fragmented architectures.
* Lower values indicate more aggregated architectures.

## Pilot Study Results

| Metric                | Zero-shot | Few-shot |
| --------------------- | --------- | -------- |
| Structural violations | 9         | 8        |
| DVR                   | 22.5%     | 20.0%    |
| DPR                   | 77.5%     | 80.0%    |
| Granularity           | 8         | 8        |

### Key Findings

* Both prompting strategies produced decompositions with the same granularity.
* The few-shot strategy resulted in fewer structural violations.
* The reduction in violations directly improved the DVR metric.
* Consequently, the DPR achieved by few-shot was higher than that obtained by zero-shot.

## Adapting the Pipeline to Other Systems

### Change the Analyzed System

Modify the constant:

```python
PETCLINIC_ROOT
```

to point to the root directory of the new project.

---

### Update the Microservice Groupings

Two alternatives are available:

1. Update the service-mapping dictionaries used in the scripts;
2. Provide new CSV files containing decompositions generated by the LLM.

---

### Adapting to Other Programming Languages

The pipeline was developed using Tree-sitter for Java.

For other languages:

1. Replace the Tree-sitter grammar accordingly;
2. Adjust the entity extraction functions;
3. Adapt the dependency extraction rules;
4. Re-execute the complete pipeline.
