# PetClinic Analysis Methodology

## Objective

Document the static analysis process used to extract and validate the dependency graph of the Spring PetClinic project.

## Scope

- Analyzed source code: `project_descriptions/PetClinic/monolith/spring-petclinic-main/src/main/java`

## Formal Graph Definition

G_real = (V, E), where V represents the set of Java classes identified in the project and E represents the set of static structural dependencies detected by the analysis. Dependencies include inheritance, JPA relationships, composition, repository usage, and dependency injection.

## Graph Modeling Criteria

### Inclusion

- Classes belonging to the analyzed project.
- Inheritance relationships.
- JPA relationships.
- Composition dependencies.
- Injection dependencies.
- Repository → Entity relationships.

### Exclusion

- External Spring Framework classes.
- JDK classes.
- Third-party library classes.
- Dependencies resolved only at runtime.
- Reflection and dynamic code generation.

External dependencies are not modeled as graph nodes.

## Executed Process

### 1. Identification of Relevant Files

- Located the main domain packages in `org.springframework.samples.petclinic`.
- Focused on model classes, controllers, repositories, and application configuration.

### 2. Extraction of Entities and Relationships

- Analyzed class and interface declarations in Java files to capture:
  - `class X extends Y`
  - `class X implements Y`
  - Typed fields and generic collections (`List<T>`, `Set<T>`)
- Searched for JPA annotations indicating entity relationships:
  - `@OneToMany`
  - `@ManyToOne`
  - `@ManyToMany`
- Examined public constructors to identify dependencies injected through parameters.

### 3. Identification of Repositories and Domain Links

- Detected `*Repository` interfaces extending `JpaRepository` or `Repository`.
- Extracted the generic entity associated with each repository (for example, `OwnerRepository extends JpaRepository<Owner, Integer>`).
- Linked controllers and formatters to repository dependencies found in constructors.

### 4. Updating the Existing Graph

- Loaded the existing graph from `petclinic_dependency_graph.graphml`.
- Added additional edges to represent:
  - inheritance and implementations
  - JPA compositions between entities
  - repository injections into controllers
  - repository-to-managed-entity relationships

### 5. Connectivity Verification

- Created a Python script (`project_descriptions/PetClinic/graph_audit.py`) to:
  - generate a timestamped copy of the GraphML file
  - export all edges to CSV
  - calculate the degree of each node
  - identify isolated nodes and nodes with degree 1

### 6. Report Consolidation

- Updated `petclinic_report.md` with:
  - summary of the methods used
  - node and edge counts
  - selected evidence
  - analysis limitations
  - generated artifacts

## Tools and Technologies Used

### Automated Static Analysis (`petclinic_static_analysis.py`)

- **Tree-sitter with Java grammar:** Automated AST (Abstract Syntax Tree) parsing of all `.java` files in the project. The parser traverses the AST to extract class/interface/enum declarations, inheritance (`superclass`), implementations (`implements`), method invocations, object instantiations, field accesses, and types used in declarations (`class_or_interface_type`, `scoped_identifier`).

- **NetworkX:** Library used to build and manipulate the directed dependency graph (DiGraph). Each class becomes a node, and each structural relationship (inheritance, composition, injection, repository usage) becomes a directed edge. The graph is exported to GraphML through `nx.write_graphml()`.

- **Pandas:** Library used to read CSV files containing class-to-service mappings generated by LLMs (few-shot and zero-shot), associating each class with a proposed microservice.

- **Heuristic Service Mapping:** Dictionaries `PACKAGE_SERVICE_MAP` and `CLASS_SERVICE_MAP` that map packages and classes to services (Client Service, Vet Service, Visit Service, Pet Service, Gateway Service). The `assign_service()` function applies this heuristic as a fallback when no CSV is available.

### Structural Violation Detection (`violation_detection.py`)

- **ViolationDetector:** Specialized Python class that loads the real graph edge CSV, loads LLM-generated decompositions from CSV files (few-shot and zero-shot), automatically detects structural violations (edges crossing different services), calculates TVD (Dependency Violation Rate), CB (Basic Cohesion = 1 - TVD), and Granularity metrics, generates a comparative Markdown report (`petclinic_violations_report.md`), and exports raw results to JSON (`petclinic_violations_analysis.json`).

### Graph Auditing (`graph_audit.py`)

- **`graph_audit.py` script:** Python script that reads the GraphML file using regex, creates a timestamped backup, exports all edges to CSV, calculates node degrees, identifies isolated nodes (degree 0) and nodes with degree 1, and generates an audit report (`petclinic_dependency_graph_audit.txt`).

### Data Formats

- **GraphML:** XML-based format for representing the dependency graph (`petclinic_dependency_graph.graphml`), compatible with tools such as yEd and Gephi.

- **CSV:** Tabular export of graph edges (`petclinic_dependency_graph_edges.csv`) with columns `source`, `target`, `type`.

- **JSON:** Export format for violation analysis results (`petclinic_violations_analysis.json`).

### Complementary Manual Analysis

- **Manual validation and correction:** During the audit process, a gap was identified in the automated extraction: the `@ManyToMany` relationship between `Vet` and `Specialty` was not correctly captured by Tree-sitter. The edge was manually added to the graph, increasing the total number of edges from 39 to 40.

## Limitations and Assumptions

- The analysis was static and limited to the source code located in `src/main/java`.
- Runtime dependencies, reflections, dynamically created beans, and test code were not included.
- The graph models static structural relationships, not execution flow or hidden transitive dependencies.
- Validation of isolated nodes and degree-1 nodes considers only the edges currently present in the graph.

## Relevant Results

- Identified isolated nodes: `CacheConfiguration`, `CrashController`, `WebConfiguration`, `WelcomeController`.
- Identified node with degree 1: `PetClinicApplication`.
- Detected graph gap: `Specialty` should be connected to `Vet` through a `@ManyToMany` relationship in `Vet.java`, and this edge was subsequently added to the graph.

## Study Characterization

This Spring PetClinic analysis constitutes a pilot study used to validate and refine the dependency graph extraction, auditing, and construction process. The procedures consolidated during this pilot will later be applied to the systems that compose the main experimental dataset of the research.

## Final Remarks

This document describes the analysis process adopted to make the work reproducible and facilitate future reviews of the PetClinic dependency graph.
