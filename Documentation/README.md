# Experimental Evaluation of the Divergence Between LLM-Generated Monolith Decompositions and Real Source-Code Structure

This repository contains an automated and reproducible pipeline for evaluating the structural adherence of microservice architectures generated by Large Language Models (LLMs) against the actual dependency structure extracted from monolithic source code.

The project was developed as part of a Master's research program in Applied Artificial Intelligence and Software Engineering at the Federal University of Campina Grande (UFCG).

---

## Overview

Recent studies have demonstrated that Large Language Models (LLMs) can generate microservice architectures directly from textual requirements. While these architectures are often semantically coherent, they may not reflect the actual structural dependencies present in the source code.

This project investigates the degree of divergence between LLM-generated microservice decompositions and the real dependency structure of monolithic systems.

The proposed pipeline automatically:

1. Extracts dependency graphs from monolithic source code through static analysis.
2. Maps classes to services according to LLM-generated architectures.
3. Detects structural violations caused by dependencies crossing service boundaries.
4. Computes quantitative metrics that measure architectural adherence.

The current pilot study uses the Spring PetClinic system and evaluates architectures generated by OpenAI o3 (GPT-4 o3) under two prompting strategies:

- Zero-Shot
- Few-Shot

---

## Research Questions

This research investigates the following questions:

### RQ1

**What is the degree of structural adherence between LLM-generated microservice architectures and the real source code?**

### RQ2

**Do prompting strategies (Zero-Shot vs Few-Shot) influence structural adherence?**

## Analysis Pipeline

The pipeline consists of four sequential modules:

```text
┌─────────────────────┐    ┌─────────────────────┐    ┌─────────────────────┐    ┌─────────────────────┐
│ Dependency          │    │ Architecture        │    │ Structural          │    │ Metrics             │
│ Extraction          │ →  │ Interpretation      │ →  │ Violation Detection │ →  │ Computation         │
│ (Real Graph)        │    │ (LLM Output)        │    │                     │    │                     │
└─────────────────────┘    └─────────────────────┘    └─────────────────────┘    └─────────────────────┘
```

### Module 1 — Dependency Extraction

Performs static analysis of the monolithic source code using Tree-sitter and Java AST parsing.

Extracted relationships include:

- Inheritance
- Interface implementation
- JPA relationships
- Dependency injection
- Repository usage
- Typed compositions

The output is a directed dependency graph represented in GraphML format.

---

### Module 2 — Architecture Interpretation

Reads microservice decompositions generated by LLMs and maps classes to services.

Architectural decompositions are provided as CSV files containing:

```csv
class,service
Owner,Client Service
Pet,Pet Service
Visit,Visit Service
```

---

### Module 3 — Structural Violation Detection

For every dependency:

```text
A → B
```

the pipeline verifies whether:

- A and B belong to the same service
- A and B belong to different services

Dependencies crossing service boundaries are classified as structural violations.

---

### Module 4 — Metrics Computation

The framework computes three quantitative metrics.

| Metric | Description |
|----------|----------|
| TVD (Dependency Violation Rate) | Percentage of dependencies crossing service boundaries |
| TPD (Dependency Preservation Rate) | Percentage of dependencies preserved inside services |
| Granularity | Number of services proposed by the architecture |

Formally:

```text
TVD = Violations / Total Dependencies

TPD = 1 − TVD
```

## Pilot Results

### OpenAI o3 — Zero-Shot vs Few-Shot

| Metric | Zero-Shot | Few-Shot |
|----------|----------|----------|
| Structural Violations | 9 | 8 |
| TVD | 22.5% | 20.0% |
| TPD | 77.5% | 80.0% |
| Granularity | 8 | 8 |

### Interpretation

Both prompting strategies preserved more than three quarters of the original monolithic dependencies within the same service boundaries.

Few-Shot prompting achieved slightly better structural adherence:

- One fewer structural violation
- Lower TVD
- Higher TPD

These results suggest that providing examples in the prompt may help the model better respect structural boundaries present in the original implementation.

## Requirements

Install the required Python libraries:

```bash
pip install tree-sitter tree-sitter-java networkx pandas
```

Alternative:

```bash
pip install tree-sitter-languages
```

## Running the Pipeline

### 1. Dependency Extraction

```bash
python Scripts/petclinic_static_analysis.py
```

Output:

- GraphML dependency graph
- Initial graph statistics

---

### 2. Graph Audit

```bash
python Scripts/graph_audit.py
```

Output:

- Timestamped GraphML backup
- CSV export of dependencies
- Isolated node detection
- Degree analysis

---

### 3. Structural Violation Detection

```bash
python Scripts/violation_detection.py
```

Output:

- Structural violation report
- JSON results
- TVD, TPD, and Granularity metrics

---

## Limitations

- The analysis is purely static.
- Runtime dependencies are not captured.
- Reflection and dynamic bean creation are not analyzed.
- Test code is excluded.
- The current pilot uses a single monolithic system (Spring PetClinic).
- Statistical generalization is limited until larger-scale experiments are completed.
