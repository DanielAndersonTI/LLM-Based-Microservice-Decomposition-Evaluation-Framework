# Experiments - Paper 2: Decomposing Monoliths into Microservices with LLMs

This repository contains an automated pipeline for evaluating the **structural adherence of microservice architectures generated by LLMs** (Large Language Models) against the actual source-code structure of monolithic systems.

The pipeline was developed in the context of a master's research project investigating the extent to which architectures generated from textual descriptions are compatible with the system's actual structural organization.

## Research Questions

- **(RQ1)** What is the degree of adherence between architectures generated by LLMs and the actual source code?
- **(RQ2)** Does the model used — OpenAI o3 or Claude 3.5 Sonnet — influence this adherence?
- **(RQ3)** Do prompting strategies — zero-shot and few-shot — impact the obtained results?

## Pipeline Overview

The pipeline consists of **4 modules** organized sequentially:

```text
┌─────────────────────┐    ┌─────────────────────┐    ┌─────────────────────┐    ┌─────────────────────┐
│   Dependency        │    │ Architecture        │    │ Mapping and         │    │ Calculation and     │
│   Extraction        │ →  │ Interpretation      │ →  │ Violation Detection │ →  │ Summarization       │
│   (real graph)      │    │ (LLM decompositions)│    │                     │    │ (metrics)           │
└─────────────────────┘    └─────────────────────┘    └─────────────────────┘    └─────────────────────┘
         │                         │                           │                           │
         ▼                         ▼                           ▼                           ▼
   .java source code        CSVs containing classes      Real graph ×              TVD, BC, and
   → GraphML graph          and proposed services        LLM architecture          Granularity
```

### Module 1 — Dependency Extraction

Traverses the source code of the monolithic system and identifies static dependencies between classes using **Tree-sitter** (AST parsing). The result is a **dependency graph** where:

- **Nodes** represent classes or modules
- **Edges** represent directional dependencies (inheritance, JPA composition, dependency injection, repository usage)

### Module 2 — Architecture Interpretation

Reads CSV files generated in previous studies (Albuquerque et al.; Pereira et al.) and extracts the class-to-service associations proposed by the LLMs.

### Module 3 — Mapping and Violation Detection

Cross-references the dependency graph with the proposed architecture. For each edge A → B, it checks whether both classes belong to the same service:

- **Same service** → consistent dependency
- **Different services** → structural violation

### Module 4 — Calculation and Summarization

Aggregates violations and computes three metrics:

| Metric | Description | Formula |
|---|---|---|
| **TVD** (Dependency Violation Rate) | Proportion of edges crossing service boundaries | violations / total edges |
| **BC** (Basic Cohesion) | Proportion of edges internal to services | 1 − TVD |
| **Granularity** | Total number of proposed services | service count |

## How to Use the Tools

### Prerequisites

```bash
pip install tree-sitter tree-sitter-java networkx pandas
```

Alternative if `tree-sitter-java` is unavailable:

```bash
pip install tree-sitter-languages
```

### 1. Static Analysis and Graph Construction

```bash
cd project_descriptions/PetClinic
python petclinic_static_analysis.py
```

**What happens:**

1. Tree-sitter parses the AST of all `.java` files in the monolith
2. NetworkX builds a directed graph (DiGraph) with classes as nodes and dependencies as edges
3. The graph is exported to `petclinic_dependency_graph.graphml`
4. Initial metrics are displayed in the terminal

**Example output:**

```text
--- PetClinic Static Analysis ---
Java files analyzed: 25
Total graph edges: 40
Service violation (edge crossing different services): 12
TVD (Service Dependency Violation Rate): 0.3000
SPR (Service Purity Rate): 0.7000
Granularity (distinct services): 5
```

### 2. Graph Auditing

```bash
python graph_audit.py
```

**What happens:**

1. Creates a timestamped GraphML backup (`petclinic_dependency_graph.YYYYMMDDHHMMSS.graphml`)
2. Exports edges to CSV (`petclinic_dependency_graph_edges.csv`)
3. Calculates the degree of each node
4. Identifies isolated nodes (degree 0) and leaf nodes (degree 1)
5. Generates an audit report (`petclinic_dependency_graph_audit.txt`)

**Example output:**

```text
copy_path ...\petclinic_dependency_graph.20260608185412.graphml
csv_path ...\petclinic_dependency_graph_edges.csv
isolated ['CacheConfiguration', 'CrashController', 'WebConfiguration', 'WelcomeController']
degree1 ['PetClinicApplication', 'Specialty']
```

### 3. Structural Violation Detection

```bash
python violation_detection.py
```

**What happens:**

1. Loads the edge CSV from the real graph
2. Loads LLM decompositions from CSV files in `csv/` (few-shot and zero-shot)
3. Detects violations: edges crossing different services
4. Calculates metrics: TVD, BC, and Granularity
5. Generates a Markdown report (`petclinic_violations_report.md`)
6. Exports raw analysis data to JSON (`petclinic_violations_analysis.json`)

**Example generated report:**

```markdown
| Metric | Few-shot | Zero-shot |
|---------|---------|----------|
| Total Edges | 40 | 40 |
| Detected Violations | 8 | 12 |
| TVD (Violation Rate) | 20.00% | 30.00% |
| BC (Basic Cohesion) | 80.00% | 70.00% |
| Granularity (# Services) | 5 | 5 |
```

## How to Adapt for Your Own Experiments

### Scenario 1: You Have a Java Project and Want to Analyze Its Dependencies

1. Copy the scripts into your working directory.
2. Edit the `PETCLINIC_ROOT` variable in `petclinic_static_analysis.py`:

```python
PETCLINIC_ROOT = (
    BASE_DIR
    / "path"
    / "to"
    / "your"
    / "project"
    / "src"
    / "main"
    / "java"
)
```

3. Run the analysis:

```bash
python petclinic_static_analysis.py
python graph_audit.py
```

### Scenario 2: You Have LLM-Generated Microservice Decompositions and Want to Evaluate Them

1. Place your decomposition CSVs in the `csv/` folder with the following columns:

```csv
class,service
Owner,Client Service
OwnerController,Client Service
Pet,Pet Service
...
```

2. Or edit the `PACKAGE_SERVICE_MAP` and `CLASS_SERVICE_MAP` dictionaries in the script.

3. Run violation detection:

```bash
python violation_detection.py
```

### Scenario 3: Your Project Is Not Java (Python, JavaScript, C#, Go, etc.)

1. Install the corresponding Tree-sitter grammar:

```bash
# Python
pip install tree-sitter-python

# JavaScript / TypeScript
pip install tree-sitter-javascript tree-sitter-typescript

# C#
pip install tree-sitter-c-sharp

# Go
pip install tree-sitter-go
```

2. Modify parser initialization in the script:

```python
# Python example
from tree_sitter import Language
import tree_sitter_python
JAVA_LANGUAGE = Language(tree_sitter_python.language())

# JavaScript example
from tree_sitter import Language
import tree_sitter_javascript
JAVA_LANGUAGE = Language(tree_sitter_javascript.language())
```

3. Adjust AST extraction functions:
   - `extract_top_level_classes()` — declaration node names vary by language
   - `collect_dependencies()` — invocation, inheritance, and composition patterns vary
   - Refer to the Tree-sitter grammar documentation for your target language

### Scenario 4: You Want to Process Multiple Projects in Batch

Create a wrapper script that iterates through projects:

```python
projects = [
    {"name": "project-a", "root": "path/to/project-a/src"},
    {"name": "project-b", "root": "path/to/project-b/src"},
]

for proj in projects:
    print(f"Processing {proj['name']}...")
    PETCLINIC_ROOT = Path(proj['root'])
    graph, class_fqns, class_defs = build_graph(PETCLINIC_ROOT)
    nx.write_graphml(graph, f"{proj['name']}_dependency_graph.graphml")
    print(f"  → {len(graph.nodes())} nodes, {len(graph.edges())} edges")
```

### Scenario 5: You Have Microservice Architectures in Textual Format (Not CSV)

Adapt Module 2 to read your format. Example using JSON:

```python
import json

with open("decomposition.json") as f:
    data = json.load(f)

service_map = {}
for service in data["services"]:
    for cls in service["classes"]:
        service_map[cls] = service["name"]
```

## Validation Pilot

The **Spring PetClinic** system was used as a pilot study to validate and refine the pipeline. Validation results are documented in:

- `project_descriptions/PetClinic/analysis_methodology.md` — detailed methodology
- `project_descriptions/PetClinic/petclinic_report.md` — report with evidence and limitations
- `project_descriptions/PetClinic/petclinic_violations_report.md` — few-shot vs zero-shot comparison

### Pilot Results

| Metric | Few-shot | Zero-shot |
|---|---|---|
| TVD | 20.00% | 30.00% |
| BC | 80.00% | 70.00% |
| Granularity | 5 services | 5 services |

**Conclusion:** The few-shot strategy achieved better structural adherence (fewer violations) than the zero-shot strategy for PetClinic.

## Limitations

- The analysis is **static** — it captures only dependencies visible in the source code.
- Runtime dependencies, reflection, dynamic beans, and test code **are not captured**.
- The graph models **static structural relationships**, not execution flow.
- For other languages, AST extraction functions must be **manually adapted**.
- The `collect_dependencies()` function uses substring matching, which may generate **false positives** for similarly named classes.
- The system sample size is limited, restricting statistical generalization.
